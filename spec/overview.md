# 台股盤後決策系統 — 現行功能規格說明書

> 版本：2026-05-07 ｜ 基礎：目前已實作的 codebase（mock 資料驅動）

---

## 一、產品定位

**目標使用者**：每日收盤後自主研究台股的個人投資者
**核心流程**：快速掃描（3–5 分鐘）→ 深入研究（不限時）兩段式漏斗
**產品定位**：盤後市場結構研究工具（Market Research Cockpit），協助使用者整理「市場狀態、結構矛盾、研究優先順序」
**產品原則**：客觀呈現市場狀態、每個數字附歷史百分位、不提供買賣建議、不輸出多空結論、不做個股推薦排名

V1 的價值不是給答案，而是把每天分散在大盤、資金、族群、個股的資料整理成可檢查的研究流程。使用者最後仍自行判斷，但不需要從零拼湊市場全貌。

### 命名與功能邊界

產品需避免任何會被理解成交易建議、選股推薦或隔日預測的功能名稱與文案。

| 避免使用 | 改用 |
|----------|------|
| 買進 / 賣出 / 進場 / 出場 | 資金結構、籌碼支撐、延續性確認 |
| 推薦個股 / AI 嚴選 / 飆股雷達 | 個股研究、條件篩選、結構異常 |
| 明日觀察清單 | 待驗證清單 / 研究清單 / 後續追蹤項目 |
| 市場劇本 | 市場環境分型 |
| 異常訊號 | 結構分歧 / 異常狀態 |
| 股票總分 / 多空評分 | 趨勢、籌碼、評價、成長四面向檢查 |

## 二、技術架構

| 層次 | 技術 |
|------|------|
| UI 框架 | React 19 + TypeScript |
| 狀態管理 | `useState`（頁面導航），TanStack Query（遠端資料） |
| 資料層 | `src/services/index.ts`（目前全部回傳 mock + setTimeout 延遲） |
| 樣式 | Tailwind CSS v4（CSS-first，無 tailwind.config.js） |
| 圖表 | Recharts |
| 測試 | Vitest（僅覆蓋純 utils，無元件測試） |

**導航模型**：無 React Router，`App.tsx` 以 `useState` 驅動三層狀態切換（主頁 → SectorPage → StockPage），切換時完全 unmount/remount，無 URL 變更。

**快取策略**：所有 TanStack Query hook 設定 `staleTime: Infinity`，mock 資料不過期。Query key 格式：`[資源名稱, id?, date]`。

**衍生資料**：`src/utils/` 放所有純計算函式（`marketBrief.ts`、`stockResearch.ts`、`percentile.ts`、`fundamental.ts`），不在 hook 或元件內做計算。

**共用研究元件**：籌碼流向、新聞列表、個股四面向檢查等跨頁重用 UI 放在 `src/components/stock/`，避免右側面板與個股頁呈現邏輯分歧。

##### 市場環境分型（Market Regime）

BlockB 應提供一個描述性 market regime label，協助使用者在 30 秒內理解今天屬於哪一種市場結構。分型只描述狀態，不代表操作建議。

| 分型 | 判斷語意 |
|------|----------|
| 擴散型上漲 | 指數上漲、廣度偏高、強勢族群數增加，資金不只集中在少數權值 |
| 集中型上漲 | 指數上漲但廣度普通或偏弱，資金集中度偏高，少數族群主導盤面 |
| 指數強廣度弱 | 指數表現強於多數個股，創高或權值貢獻高，但上漲家數比例不足 |
| 資金輪動 | 流入與流出族群都明確，強弱族群切換快速，前期強勢族群退潮 |
| 弱勢反彈 | 指數或族群短線反彈，但量能、法人、廣度未同步改善 |
| 全面轉弱 | 指數下跌、廣度偏低、弱勢族群增加，資金流出範圍擴大 |
| 結構分歧 | 指標互相矛盾，沒有單一清楚方向，需回到族群與個股層檢查 |

範例文案：

> 今日屬於「集中型上漲」：指數上漲但資金集中於半導體與 AI 伺服器，整體廣度未同步擴散。

##### 結構分歧規則

結構分歧是本產品的研究亮點，優先指出「價格、資金、廣度、量能、基本面」彼此不一致的地方。

| 類型 | 顯示描述 |
|------|----------|
| 指數漲但廣度弱 | 指數上漲，但上漲家數比例偏低，盤面由少數權值或族群支撐 |
| 族群漲但法人賣 | 族群價格表現偏強，但法人淨流出，延續性需回到個股籌碼確認 |
| 法人買但價格不動 | 法人買超增加，但價格未同步反映，可能存在上檔壓力或吸籌狀態 |
| 價格強但量縮 | 漲幅或 RS 偏強，但成交量低於近期均量，追價意願不足 |
| 基本面佳但籌碼弱 | 財務品質或成長數據佳，但法人或主力未同步支持 |
| 籌碼強但評價高 | 法人或大單集中度偏強，但 P/E、P/B 或融資風險偏高 |
| 資金入但走勢弱 | `fundInWeak`：資金流入但族群走勢偏弱，需追蹤是否只是低檔承接 |
| 技術強無法人 | `techStrongNoFund`：技術面偏強，但法人買超偏弱，籌碼支撐存疑 |

## 四、百分位顏色系統

```
veryLow  → 藍色  (text-blue-400 / bg-blue-900)
low      → 淡藍  (text-blue-300)
mid      → 灰色  (text-zinc-400)
high     → 橙色  (text-orange-400)
veryHigh → 紅色  (text-red-400)
```

漲跌方向：上漲 = 紅（台股慣例），下跌 = 藍。

### 百分位語意規則

百分位只代表「目前數值在歷史區間中的位置」，不直接代表好壞。不同指標需依投資語意解讀：

| 類型 | 指標 | 解讀方式 |
|------|------|----------|
| 越高通常越強 | RS、上漲家數比例、ROE、EPS YoY、營收 YoY | 高百分位代表相對強勢或品質改善 |
| 越高通常風險越高 | 融資使用率、當沖比、P/E、P/B、券資比 | 高百分位代表擁擠、昂貴或短線風險升高 |
| 需情境判讀 | 成交量、法人買賣超、資金集中度、大單集中度 | 需搭配價格、廣度與趨勢判斷 |

UI 顯示中文標籤時應避免把「偏低」直接寫成「便宜」或「安全」，除非該指標已明確定義為評價類指標。

## 七、AI 結構洞察與資料可信度

### AI 結構洞察原則

AI 不作為固定摘要區塊，也不重述使用者已能從圖表快速看到的數字。V1 不顯示「為了摘要而摘要」的 AI 文字；只有當 AI 能指出跨維度關係或結構分歧時，才作為洞察補充出現。

AI 洞察不是投資建議層。每一句輸出都應能回溯到明確資料依據，避免看起來像自由評論。

| 要求 | 說明 |
|------|------|
| 可追溯 | 句子需包含對應指標，例如 RS、法人買賣超、廣度、量能、百分位 |
| 有洞察才顯示 | 若只是在重述已呈現數字，整個 AI 區塊不顯示 |
| 跨維度 | 優先指出價格、資金、廣度、籌碼、評價、成長之間不一致或同步改善的關係 |
| 不預測 | 不使用「明天會漲」、「即將反攻」等預測語句 |
| 不建議 | 不使用「應買進」、「應賣出」、「可進場」等操作語句 |
| 指出分歧 | 優先描述資料不一致處，而不是只重述排行榜 |
| 失敗 fallback | LLM 失敗時顯示最近一次快取評語與產生時間 |

建議文案格式：

> 半導體族群近 20 日 RS 上升，法人連 5 日買超，族群廣度維持 70% 以上，動能結構相對完整。

更高價值的洞察格式：

> 伺服器零組件未進入今日資金流入 Top3，但近 20 日 RS 連續上升、法人賣壓收斂，屬於需要後續驗證的結構改善。

避免文案格式：

> 半導體族群後續看漲，可以積極留意。

### 資料可信度與更新時間

真實 API 接入後，每個高影響資料區塊需顯示資料日期或更新時間，避免盤後研究誤用延遲資料。

| 資料 | UI 要求 |
|------|---------|
| 大盤與族群價量 | 顯示交易日期 |
| 法人買賣超 | 顯示資料日期，若交易所尚未更新需標示 |
| 基本面 | 顯示財報期別或最近更新月份 |
| 新聞 | 顯示來源與發布時間 |
| AI 結構洞察 | 顯示產生時間，快取內容需標示「快取」或「上次更新」；無跨維度洞察時不顯示 |

## 八、優缺點分析

### 優點

**1. 研究漏斗設計清晰**
主頁 → 族群頁 → 個股頁，每一層都有明確的「回答哪個問題」的責任切分。使用者不會迷失在資訊裡。

**2. 資料層解耦徹底**
`src/services/index.ts` 是唯一與資料打交道的地方，換成真實 API 不需改元件或 hook，只要改這一個檔案。

**3. 衍生邏輯集中，可測試**
`buildMarketBrief()`、`getStockRank()`、`buildStockChecks()`、`getPercentileColor()`、`fundamentalValueColor()` 都是純函式，方便寫單元測試，目前已有 market brief、stock research、fundamental 相關測試覆蓋。

**4. 百分位系統統一，但保留指標語意**
所有指標都透過同一套 `PercentileLabel` 機制輸出歷史位置，同時依指標類型區分「強弱」、「風險」、「情境判讀」，避免把百分位誤解成單純好壞。

**5. 獨立錯誤邊界**
各 Block 錯誤互不干擾，避免一個 API 壞掉讓整頁白屏。

**6. 型別安全**
`src/types/index.ts` 單一來源，TypeScript strict mode，資料流有完整型別保護。

**7. 具備專業研究差異化**
市場環境分型、結構分歧與個股四面向檢查能讓產品從資訊看板升級為盤後研究 cockpit。

**8. 右側面板與個股頁同源**
族群排名、四面向檢查、籌碼流向、新聞列表已抽到共用純函式與共用元件，降低不同頁面顯示不一致的風險。

---

### 缺點

**1. 導航無 URL，無法分享、無法上下頁**
切換族群或個股不會改 URL，使用者無法把特定個股頁存書籤，也無法用瀏覽器上一頁回到族群頁（會回到主頁）。

**2. `Stock` 型別太胖**
今日欄位、區間表現、籌碼風險三組資料全部合在一個 `Stock` 型別，造成每次取族群個股清單都要傳三組資料，後端 API 設計被型別牽制。

**3. `StockPage` 仍需查詢族群上下文**
`StockPage` 目前仍透過 `useSectors` + `useSectorStocks` 取得族群名稱、族群分類與排名所需資料。排名計算已改用共用純函式，且 TanStack Query 會吃既有快取，但未來若導入 URL router，仍可考慮由路由狀態或 detail API 一次帶入完整研究上下文。

**4. 個股研究（StockScreenerPanel）未與主導航整合**
主頁 BlockD 的「個股研究」Tab 可以直接跳進個股頁，但當使用者在個股頁按上一頁，只會回到主頁而非回到篩選狀態，研究脈絡斷裂。

**5. 目前全部是 mock 資料**
所有 API 都回傳靜態假資料 + setTimeout，無任何真實後端接入，無法驗證實際的資料延遲、分頁、格式邊界情況。基本面資料（P/E、P/B、ROE 等）的百分位計算邏輯尚未經後端確認。

**6. 元件層級無測試**
測試只覆蓋純 utils，所有 React 元件（包含複雜的互動邏輯如 BlockE 三 Tab 排序）都沒有測試，重構風險高。

**7. 消費者化功能需避免刺激性命名**
若後續增加快速入口層，應使用「市場環境分型」、「研究清單」、「結構分歧」、「後續追蹤項目」等研究語言，不使用「明日觀察清單」、「市場劇本」、「AI 嚴選」、「飆股雷達」等容易被理解為交易提示的命名。

**8. 市場環境分型仍需真實資料驗證**
目前 BlockB 已有 rules-based regime label、結構分歧與個股四面向檢查，但仍需真實 API 與歷史資料驗證規則穩定性。
